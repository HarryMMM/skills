# 新特性开发流程

## 适用场景
在现有工程中新增业务能力、接口、工具方法或流程编排。

## 引导式交互原则
- 每个步骤只问一个问题或一组紧密相关的问题
- 等待用户回答后再继续下一步
- 在关键决策点给出推荐选项
- 每完成一个阶段，向用户反馈进度

## 执行步骤

### 1. 开场说明
向用户说明接下来需要：
- 了解要开发的特性需求
- 分析影响范围与落点文件
- 制定实现方案并等待确认
- 实现功能并验证

### 2. 需求分析
**依次询问：**

**问题 1：特性目标**
> 请描述要新增的特性功能是什么？要解决什么问题？

**问题 2：输入输出**
> 这个特性的输入参数和返回值是什么？是否有副作用？

**问题 3：边界条件**（可选）
> 是否有特殊的异常场景、性能约束或安全要求？

### 3. 影响范围分析
根据需求分析结果，识别影响范围与落点文件：

**向用户展示分析结果：**

| 代码类型 | 放置目录 | 具体文件 |
|---|---|---|
| 对外入口/编排 | `app/tools/` 或 `app/apis/` | 新增 xxx_tool.py |
| 业务逻辑 | `app/services/` | 新增 xxx_service.py |
| 数据访问 | `app/repositories/` | 新增 xxx_repository.py |
| 数据模型 | `app/models/` | 新增 xxx_model.py（如需） |

> 以上是实现方案，是否继续？

**等待用户确认后才能执行后续步骤。**

### 4. 实现
**向用户反馈：**
> 正在实现功能...

按分层约束的顺序实现：
1. **repositories**：先实现数据访问层，确保 SQL 参数化
2. **services**：实现业务逻辑，调用 repository 方法
3. **tools/apis**：实现对外入口，调用 service 方法

如涉及配置项：
- 在 `app/config/settings.py` 中补充字段定义
- 在 `.env.example` 中补充示例值

如涉及模型：
- 在 `app/models/` 中定义数据模型

### 5. 测试
- 关键行为补充单元测试（`tests/` 目录）
- 至少提供可运行的冒烟验证样例

### 6. 执行验证
**向用户反馈：**
> 正在执行自动验证...

运行 `python scripts/run_validation.py <project_root>`，依次检查：
1. 结构检查 — 目录/文件存在性
2. 配置检查 — pyproject.toml / .env.example 关键配置
3. 分层/风格检查 — apis/tools/services 分层约束、print 检测
4. 编译/导入冒烟检查

输出固定三段：自动验证通过 / 自动验证失败或受阻 / 需要人工确认。
未执行的检查不能算通过，脚本执行失败时标记为"受阻"。

### 7. 输出结果
**向用户展示最终结果：**

1. **特性实现摘要** — 实现了什么功能
2. **变更文件清单** — 列出所有新增/修改的文件
3. **兼容性与风险说明** — 对现有功能的影响
4. **自动验证结果** — 通过/失败/受阻项
5. **人工复核清单**（如有）— 需要用户手动确认的事项

## 约束
- 用户未确认前不得执行实现操作
# 新特性开发流程

## 适用场景
在现有工程中新增业务能力、接口、工具方法或流程编排。

## 执行步骤

### 1. 需求分析
确认以下信息：
- **目标**：要解决什么问题
- **输入输出**：参数、返回值、副作用
- **边界条件**：异常场景、性能约束、安全要求

### 2. 用户确认
将上一步的需求分析结果展示给用户，等待用户确认无误后才能继续。用户未确认前不得执行后续步骤。

### 3. 落点规划
根据分层约束确定文件归属：

| 代码类型 | 放置目录 | 示例 |
|---|---|---|
| 对外入口/编排 | `app/tools/` 或 `app/apis/` | 新增工具方法、REST 接口 |
| 业务逻辑 | `app/services/` | 新增业务处理类 |
| 数据访问 | `app/repositories/` | 新增数据查询 |
| 数据模型 | `app/models/` | 新增模型定义 |

### 4. 实现
按 `tools → services → repositories` 的顺序实现：
1. **repositories**：先实现数据访问层，确保 SQL 参数化
2. **services**：实现业务逻辑，调用 repository 方法
3. **tools**：实现对外入口，调用 service 方法

如涉及配置项：
- 在 `app/config/settings.py` 中补充字段定义
- 在 `.env.example` 中补充示例值

如涉及模型：
- 在 `app/models/` 中定义数据模型

### 5. 测试
- 关键行为补充单元测试（`tests/` 目录）
- 至少提供可运行的冒烟验证样例

### 6. 执行验证
运行 `python scripts/run_validation.py <project_root>`，依次检查：
1. 结构检查 — 目录/文件存在性
2. 配置检查 — pyproject.toml / .env.example 关键配置
3. 分层/风格检查 — apis/tools/services 分层约束、print 检测
4. 编译/导入冒烟检查

输出固定三段：自动验证通过 / 自动验证失败或受阻 / 需要人工确认。
未执行的检查不能算通过，脚本执行失败时标记为“受阻”。

### 7. 输出结果
1. 特性实现摘要
2. 变更文件清单
3. 兼容性与风险说明
4. 自动验证结果
5. 人工复核清单（如有）
