# 新特性开发流程

## 适用场景
在现有工程中新增业务能力、接口、工具方法或流程编排。

## 执行步骤

### 1. 需求分析
确认以下信息：
- **目标**：要解决什么问题
- **输入输出**：参数、返回值、副作用
- **边界条件**：异常场景、性能约束、安全要求

### 2. 用户确认
将上一步的需求分析结果展示给用户，等待用户确认无误后才能继续。用户未确认前不得执行后续步骤。

### 3. 落点规划
根据分层约束确定文件归属：

| 代码类型 | 放置目录 | 示例 |
|---|---|---|
| 对外入口/编排 | `app/tools/` 或 `app/apis/` | 新增工具方法、REST 接口 |
| 业务逻辑 | `app/services/` | 新增业务处理类 |
| 数据访问 | `app/repositories/` | 新增数据查询 |
| 数据模型 | `app/models/` | 新增模型定义 |

### 4. 实现
按 `tools → services → repositories` 的顺序实现：
1. **repositories**：先实现数据访问层，确保 SQL 参数化
2. **services**：实现业务逻辑，调用 repository 方法
3. **tools**：实现对外入口，调用 service 方法

如涉及配置项：
- 在 `app/config/settings.py` 中补充字段定义
- 在 `.env.example` 中补充示例值

如涉及模型：
- 在 `app/models/` 中定义数据模型

### 5. 测试
- 关键行为补充单元测试（`tests/` 目录）
- 至少提供可运行的冒烟验证样例

### 6. 执行验证
运行 `python scripts/run_validation.py <project_root>`，依次检查：
1. 结构检查 — 目录/文件存在性、禁止目录
2. 配置检查 — pyproject.toml / .env.example 关键配置
3. 分层/风格检查 — apis/tools/services 分层约束、print 检测
4. 编译/导入冒烟检查

输出固定三段：自动验证通过 / 自动验证失败或受阻 / 需要人工确认。
未执行的检查不能算通过，脚本执行失败时标记为“受阻”。

### 7. 输出结果
1. 特性实现摘要
2. 变更文件清单
3. 兼容性与风险说明
4. 自动验证结果
5. 人工复核清单（如有）
