# 重构工程流程

## 适用场景
用户希望将现有 Python 项目重构为标准结构。

## 引导式交互原则
- 每个步骤只问一个问题或一组紧密相关的问题
- 等待用户回答后再继续下一步
- 在关键决策点给出推荐选项
- 每完成一个阶段，向用户反馈进度

## 执行步骤

### 1. 开场说明
向用户说明接下来需要：
- 扫描现有项目结构
- 识别与标准结构的差异
- 制定重构方案并等待确认
- 执行重构并验证

### 2. 扫描现状
**向用户反馈：**
> 正在扫描项目结构...

- 列出项目根目录的所有子目录和关键文件
- 识别现有结构类型（flat / src-layout / 其他）
- 检查是否已有 `pyproject.toml`、`.env.example`

### 3. 差异识别与方案展示
对照 SKILL.md 中的标准目录结构，逐项对比：
- 缺失目录：需要新建
- 多余目录：评估是否可迁移或删除
- 文件错位：需要移动到正确位置
- 配置缺失：需要补充模板文件

**向用户展示差异清单：**

| 类型 | 项目 | 建议操作 |
|---|---|---|
| 缺失 | app/config/ | 新建 |
| 错位 | src/main.py | 移动到 app/main.py |
| 多余 | old_utils/ | 建议删除 |

> 以上是扫描结果与建议操作，是否继续执行重构？

**等待用户确认后才能执行后续步骤。**

### 4. 执行改造（按顺序）

**向用户反馈：**
> 正在执行重构...

**阶段 A：结构调整**
- 创建缺失目录和 `__init__.py`
- 移动错位的文件到正确目录

**阶段 B：基础设施**
- 生成/更新 `pyproject.toml`（从模板）
- 生成/更新 `Dockerfile`、`.env.example`、`README.md`
- 生成/更新 `.gitignore`

**阶段 C：分层代码**
- 将数据库访问代码迁移到 `repositories/`
- 将业务逻辑迁移到 `services/`
- 将对外入口迁移到 `tools/` 或 `apis/`
- 更新所有 import 路径

### 5. 兼容性处理
- 保持原有公共 API 签名不变
- 如原项目有入口脚本，保留并改为调用新的 `app.main`
- 不改变无关业务行为

### 6. 执行验证
**向用户反馈：**
> 正在执行自动验证...

运行 `python scripts/run_validation.py <project_root>`，依次检查：
1. 结构检查 — 目录/文件存在性
2. 配置检查 — pyproject.toml / .env.example 关键配置
3. 分层/风格检查 — apis/tools/services 分层约束、print 检测
4. 编译/导入冒烟检查

输出固定三段：自动验证通过 / 自动验证失败或受阻 / 需要人工确认。
未执行的检查不能算通过，脚本执行失败时标记为"受阻"。

### 7. 输出结果
**向用户展示最终结果：**

1. **差异摘要** — 改造前 vs 改造后对比
2. **改动文件清单** — 列出所有变更的文件
3. **兼容性说明** — 哪些行为保持不变，哪些可能有影响
4. **自动验证结果** — 通过/失败/受阻项
5. **人工复核清单** — 需要用户手动确认的事项

## 约束
- 不为"美化"引入无关抽象
- 验证失败项必须明确说明原因
- 用户未确认前不得执行重构操作
